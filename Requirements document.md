# 任务目标

请根据以下需求，完成一个**可扩展的 CAN 总线分析工具（类似 ZCANPRO）的硬件驱动端设计**，并输出完整的企业级设计文档和硬件代码。**App 端暂不实现**，只实现硬件端（固件）。

## 核心需求

### 1. 多 MCU 支持与可扩展性
- 必须支持 **STM32F103C8T6** 和 **STM32F407VET6**。
- 未来可轻松添加其他 MCU（如 STM32H7、GD32 等），新增芯片时**只需在 `device/` 目录下增加该芯片的配置文件**（如 `device/stm32f103_config.h`, `device/stm32f407_config.h`），**无需修改核心驱动代码**。
- 代码结构必须实现**硬件抽象层（HAL）**，将 MCU 外设（USART、USB、CAN、ADC）与业务逻辑彻底分离。

### 2. 硬件功能要求
- **CAN 总线探测**：
  - 支持发送报文和接收监听报文（含高精度时间戳，如 1μs 或 1ms 分辨率）。
  - 支持错误帧检测与上报（CRC 错误、位填充错误、应答错误、帧格式错误等）。
  - 即使没有连接 ADC 硬件（或无 ADC 外设），也必须**支持普通总线探测**（仅通过 CAN 控制器本身的接收功能）。当 ADC 存在且启用时，可提供高精度波形采样；当 ADC 不存在或未启用时，硬件仍能完成 CAN 报文收发与监听。
- **波形采集**：
  - 支持可选的 **ADC 高精度波形采集**（用于波形显示，如 CAN 总线电平或外部模拟信号）。
  - 硬件端需要提供接口供 App 查询是否支持 ADC 波形采集（能力查询机制）。
  - 当 App 请求“总线波形探测”时，硬件端应能自动判断当前是否启用 ADC：
    - 若 ADC 已初始化且可用，则返回 ADC 波形数据。
    - 若 ADC 不可用，则返回普通探测结果（例如仅 CAN 报文时间戳和电平逻辑值，或返回不支持波形提示）。
  - 留给 App 的接口应允许明确选择“波形来源”（ADC 模式 / 普通模式），同时硬件提供能力查询接口。
- **通信接口**：
  - 与 App 的通信方式：**串口（USART）** 和 **USB（CDC 虚拟串口）** 均可选，硬件端需同时支持两种物理接口，可通过配置文件或运行时选择主通信通道。
  - 支持 **自动设备识别**：硬件上电后主动发送设备描述信息（如通过串口/USB 发送特定标识帧），App 端可据此发现设备。

### 3. 硬件端启动与初始化流程
- **当 App 尚未发送任何连接或配置指令时**，硬件端应自动完成以下必要初始化：
  - 时钟、GPIO、CAN 外设（默认关闭接收/发送，或进入静默模式）、串口/USB 通信接口（使能，准备接收命令）。
  - ADC（如果配置了且硬件支持）可选择是否默认初始化，但需提供开关（可通过编译配置或运行时命令启用）。
  - **不要**主动开始 CAN 报文监听或波形采样，只等待 App 的进一步指令。
- 初始化完成后，硬件应处于“待命”状态，能够响应 App 的探测命令（如 `GET_INFO`、`GET_CAPABILITIES` 等）。

### 4. 硬件端 API 设计规范（丰富且硬件分离）
- 硬件端需提供**丰富的内部 API 函数**，供应用层（即协议处理模块）调用。这些 API 必须基于硬件抽象层，不依赖具体的 MCU。
- **规范要求（不限定具体函数名）**：
  - 必须包含 CAN 控制相关的 API：初始化、配置波特率/工作模式、发送 CAN 帧、接收 CAN 帧（含阻塞/非阻塞支持）、获取错误状态、清空错误计数器等。
  - 必须包含 ADC 控制相关的 API：初始化（采样率、分辨率、通道）、启动采样、停止采样、读取采样数据、查询 ADC 是否可用等。
  - 必须包含通信收发相关的 API：通过选定的物理接口（串口/USB）发送原始字节、接收原始字节，并支持接口切换。
  - 必须包含设备信息与能力查询的 API：获取 MCU 型号、固件版本、支持的功能位图（是否支持 ADC、CAN 通道数、最高采样率等）。
  - 所有 API 必须保持**硬件无关**：上层协议处理模块调用时不依赖特定 MCU 的寄存器。
  - **禁止**在 API 内部混入 App 端的逻辑（如解析特定命令帧），API 只提供纯粹的硬件操作能力。
  - 每个 API 必须有清晰的输入输出定义、错误码返回，并带详细注释。
- 具体函数命名、参数结构体由实现者自行设计，但需遵循以上规范。

### 5. 与 App 的通信协议规范
- 设计一套**精简、完整、无歧义**的通信协议，用于 App ↔ 硬件之间的命令与数据交互。
- 协议要素：
  - 帧头（固定字节 + 长度）
  - 命令码
  - 数据域（可变）
  - 校验（如 CRC8 或 CRC16）
  - 时间戳（在接收到的 CAN 报文中必须包含硬件产生的时间戳）
- 必须涵盖以下命令类别（具体命令码由实现者定义）：
  - **信息查询类**：固件版本、MCU 型号、能力集（是否支持 ADC、支持的最高采样率、CAN 通道数等）
  - **参数配置类**：CAN 波特率、工作模式、滤波设置、波形采样率
  - **控制类**：启动/停止 CAN 监听、启动/停止波形采集（需区分 ADC 模式或普通探测模式）、查询 ADC 状态
  - **数据发送类**：发送单帧或多帧 CAN 报文
- 硬件 → App 的消息必须包括：
  - **CAN 报文上传**（含时间戳、帧数据、错误标记）
  - **波形数据上传**（ADC 采样值或普通探测的电平数据包）
  - **错误通知**（CAN 错误、通信错误、ADC 异常等）
  - **设备心跳/自动识别帧**（上电后主动发送）
- 协议设计必须做到**硬件分离**：App 不需要知道硬件是 F103 还是 F407，仅通过协议与硬件交互。
- app 会通过配置文件判断目前支持那些 MCU
