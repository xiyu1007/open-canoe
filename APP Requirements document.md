# 任务目标

请根据以下需求，完成一个 **CAN 总线分析工具的上位机软件（类似 ZCANPRO）**，并与现有硬件驱动端对接。硬件驱动端已完成（STM32F103/F407，支持 CAN 报文收发、ADC 波形采集、串口/USB 通信），现需实现完整的桌面应用程序（App）。

## 核心需求

### 1. 功能模块

#### 1.1 设备管理与烧录
- **当前连接芯片显示**：在 App 界面显著位置显示当前连接的硬件型号（如 `STM32F103C8T6` / `STM32F407VET6`），并实时提示连接状态（已连接/断开）。
- **自动设备识别**：App 启动后自动扫描可用通信接口（串口/USB），识别为支持的硬件后自动连接，并显示芯片型号。
- **芯片烧录选项**：
  - 提供烧录功能入口（按钮或菜单）。
  - 支持选择固件文件（.bin/.hex），显示烧录进度。
  - 烧录完成后自动识别新固件并刷新连接状态。
  - 烧录方式不限（可调用外部工具或通过硬件接口实现），但需保证可扩展性。

#### 1.2 CAN 报文显示与发送
- **CAN 报文显示区域**：
  - 以表格形式显示：时间戳、CAN ID（区分标准/扩展帧）、数据（HEX 格式）、帧类型、数据长度、错误标记等。
  - 支持颜色标记错误帧（如红色背景）。
  - 支持过滤显示（按 ID 过滤、仅显示错误帧等）。
  - 支持暂停/恢复显示、清空显示、导出为 CSV/TXT 文件。
- **CAN 报文发送功能**：
  - **单次发送**：用户输入 CAN ID、数据（HEX）、选择帧类型（标准/扩展），点击按钮立即发送一帧。
  - **循环发送**：可配置发送间隔时间（ms）、发送次数（或无限循环），支持启动/停止。
  - 支持预设报文管理（保存/加载常用报文配置）。
- **报文统计**：实时显示总发送帧数、接收帧数、错误帧数。

#### 1.3 波形显示窗口
- **独立窗口**：可通过 `View` 菜单打开/关闭。
- **数据来源**：支持两种模式：
  - **普通探测模式**：显示 CAN 总线逻辑电平的简易波形。
  - **高精度 ADC 模式**：显示 ADC 采样的真实模拟波形（需硬件支持）。
- **功能要求**：
  - 支持实时波形刷新。
  - 支持缩放、拖动、测量光标等基本波形操作。
  - 当硬件不支持 ADC 时，自动禁用高精度模式并提示用户。
  - 提供启动/停止波形采集的控制按钮。

#### 1.4 环回模式测试支持
- **硬件环回模式**：
  - App 需提供**环回模式（Loopback Mode）开关**，用于在不连接外部 CAN 总线的情况下进行自测。
  - 开启环回模式后，硬件发送的 CAN 报文会被自身接收，App 可在报文显示区域看到发送的报文被回环接收。
  - 环回模式应同时支持 **CAN 报文收发测试** 和 **波形测试**（环回模式下波形数据由硬件模拟生成或基于发送数据生成）。
- **测试用途**：环回模式用于验证 App 与硬件的通信链路是否正常，以及 CAN 功能的基本可用性，无需依赖外部 CAN 节点。

#### 1.5 面板管理
- 所有主要功能面板（发送面板、统计面板、连接设置面板等）应设计为**可关闭/可打开**的独立面板。
- 用户可通过 `View` 菜单控制各面板的显示/隐藏。
- App 应记忆用户的面板布局偏好，下次启动时恢复。

#### 1.6 设置与国际化
- **设置窗口**：包含以下可配置项：
  - 通信接口参数（波特率、数据位等）
  - CAN 总线参数（仲裁段波特率、工作模式、环回模式开关等）
  - 波形采样率设置
- **中英文切换**：
  - 在 `Settings` 菜单或设置窗口中提供语言选项。
  - 切换后界面所有文字（菜单、按钮、标签、对话框等）即时替换为对应语言。
  - **注意**：不是自动翻译软件，而是应用内预置中英文字符串资源，手动切换。
- **主题切换**（可选但建议）：支持暗色/亮色主题切换，提升视觉体验。

### 2. 与硬件驱动端的交互要求
- **通信协议**：遵循硬件设计文档中定义的通信协议（二进制帧，含命令码、数据域、校验等）。App 端需完整实现协议编解码。
- **能力查询**：App 启动时查询硬件能力（是否支持 ADC、最高采样率、CAN 通道数、是否支持环回模式等），并根据查询结果禁用/启用对应的 UI 功能。
- **环回模式控制**：App 需能通过协议命令开启/关闭硬件的环回模式。
- **错误处理**：
  - 硬件上报的错误帧需在 UI 上明确标识。
  - 通信超时、设备断开等异常需在界面上提示用户。
  - 提供自动重连机制。
- **硬件接口适配**：如果现有硬件驱动端接口不完善（如缺少环回模式命令、ADC 状态查询命令、波形数据打包格式不明确），请**同时提供修改后的硬件接口代码**。修改需遵循原有硬件架构（`device/` 配置方式），**不破坏硬件扩展性**。

### 3. UI/UX 设计要求
- **美观是第一优先级**：界面需现代化、专业感，色彩搭配协调，控件布局合理，间距、圆角、阴影等细节处理到位。
- **布局建议**：
  - 顶部菜单栏和工具栏。
  - 可侧边停靠/浮动的功能面板（发送、统计、设置等）。
  - 中央主要区域为 CAN 报文表格。
  - 底部状态栏显示连接状态、芯片型号、收发计数等信息。
- **交互反馈**：
  - 按钮有 hover/点击效果。
  - 长时间操作（烧录、循环发送）需显示进度指示。
  - 错误提示清晰但不突兀。
- **图标**：使用合适的图标库，保持视觉一致性。

### 4. 性能要求
- CAN 报文接收显示流畅，界面不卡顿。
- 波形显示实时（≥30fps 刷新率）。
- 长时间运行内存稳定，无泄漏。
- 大数据量报文（如每秒数千帧）时表格更新不阻塞 UI。

### 5. 代码质量要求
- **代码简洁无冗余**：避免重复逻辑，公共部分抽取复用。
- **结构清晰**：业务逻辑与界面分离，硬件通信与 UI 解耦。
- **可扩展性**：未来增加新硬件或新功能时，只需添加对应模块，不改动核心代码。
- **规范性**：代码有适当注释，命名清晰，遵循主流编码规范。
- **路径规范**：项目中所有涉及文件路径的地方（如配置文件、日志、导出文件、固件烧录文件等）**必须使用相对路径**，确保项目在不同目录下可移植。

### 6. 测试要求（重要）

#### 6.1 测试流程
在完成 App 代码编写后，你需要**模拟完整的硬件连接环境**，对每个功能模块进行测试。由于无法连接真实硬件，请在代码中**内置一个硬件模拟器（Mock Hardware）**，该模拟器：
- 模拟 STM32F103C8T6 和 STM32F407VET6 的行为。
- 支持环回模式：发送的 CAN 报文会被模拟器自动接收并返回给 App。
- 支持波形数据模拟：在环回模式下生成模拟波形数据。
- 支持能力查询响应（报告支持 ADC、环回模式等）。
- 支持烧录模拟（仅模拟进度，不实际烧录）。
